한 줄 요약

Shape Up으로 스프린트마다 설계(분석·기획·디자인)와 개발(빌드·QA)을 분리해 관리

7~8월 틀 만들기 — 셸·SDS·온보딩→ 9~10월 본체+데이터 — 화면군·인사이트 병행→ 11월 진입경험 — 튜토리얼·10분세팅

리스크 한눈에 — 일이 두 번 몰린다

① 9~10월 설계가 몰린다 — S4·S5를 같이 그려야 하는데 골격을 나 혼자 잡음. → 인사이트 골격을 8월 말로 당겨 분산.

② 11~12월 개발이 몰린다 — 설계는 10월에 끝나도 개발이 뒤에 붙어 S4·S5·S6 빌드 + 통합 QA가 분기 말에 겹침. → 낮은 우선순위 화면 4Q 이월, 12월은 QA 여력 확보.

나머지(IA 합의·온보딩 초과·임베드 지연·화면군 폭발)는 모두 "필수만 3Q, 나머지 4Q 이월"로 막는다.

0이번 3Q 디자인 스코프

대통합 = 흩어진 5개 제품을 하나의 콘솔로 합치는 전술. 카탈로그상 3개 유저스토리(U1~U3)로 구성.

2스프린트 정책 — Shape Up

가치덩어리 크기에 맞춰 2주 / 4주 / 6주 혼합. 각 스프린트를 설계(shaping) → 개발(building) 두 트랙으로 분리해 관리한다.

설계 기간

= 분석 + 기획 + 디자인. 설계자(기획자겸 디자이너) 담당. 이 기간이 끝나는 시점이 개발 핸드오프.

개발 기간

= 빌드 + 개발 AC 정합 + QA. 환경은 테스트 → 스테이지 → 프로드. ① 개발자가 테스트 서버에서 QA → ② 설계자가 스테이지 서버에서 최종 확인 → ③ 통과 시 프로드 배포. 주수는 추정치 ⚠️ (개발팀 확정 필요).

정석은 설계 선행 → 개발 후행이되, 핸드오프 직전 설계 후반부와 개발 초반이 1~2주 겹친다(개발팀과 논의하며 스펙 조정되는 구간). 설계가 완전히 끝나길 기다리지 않고 확정된 부분부터 개발이 먼저 붙는다.
SDS(토큰·컴포넌트·폴리시)는 별도 스프린트가 아니라 전 구간 상시 병행 트랙 — 실사용 화면에서 등장하는 컴포넌트를 기준으로 킷이 결정되므로, 7월 최소 시드 후 각 스프린트와 함께 계속 확장. 단 브랜딩·로그인은 시각적 완결성이 필요해 별도 2주(S2)로 분리.

4분기 타임라인 — 설계 / 개발 분리

각 스프린트를 설계 기간(분석·기획·디자인)과 개발 기간(빌드·AC정합·QA)으로 쪼갬. 개발은 설계 후반과 1~2주 겹쳐 착수. 개발 주수는 추정치 ⚠️.

설계 병목 · 9~10월

S4(제품 화면군)와 S5(인사이트 표면)를 같은 시기에 그려야 하는데, 두 화면의 골격을 둘 다 나 혼자 잡는다. 한 사람에게 두 덩어리가 겹치니 손이 모자라는 구간.

대응 — 인사이트 골격을 8월 말로 미리 당겨 일부 분산, 9~10월엔 스쿼드 편입 검수에 집중.

개발 병목 · 11~12월

설계는 10월에 끝나지만 개발은 그 뒤에 붙는다. 그래서 S4·S5·S6 빌드가 한꺼번에 11~12월로 밀려 몰리고, 분기 말 통합 QA까지 겹친다.

대응 — 낮은 우선순위 화면 개발은 4Q로 이월, 설계는 이 기간 QA·스펙 대응 여력 확보.

흐름 — 설계 트랙: 7~8월 셸·SDS·온보딩 → 9~10월 화면군·인사이트 병행 → 11월 튜토리얼. 개발 트랙: 7월 말부터 핸드오프 따라 흐르되 빌드 꼬리가 분기 말로 밀려 12월 3~4주는 통합 QA + 버퍼.

5가치덩어리별 상세

설계 단계별 소요 일수와 개발 핸드오프 의존성.

S1. 콘솔 셸·IA 합의

설계 2주 · 7월 1~2주 · U1

AC "레일+단 LNB, 순환의존 0". 그릇 확정 없이는 전부 흔들림.

분석 3일기획 4일디자인 3일개발 2주
  • 5개 제품 IA 전수 매핑, 중복/충돌 메뉴 식별, 데모2 레일+LNB 역설계
  • 레일+LNB 구조, 공통 프리미티브(NavItem/Section/Badge) 정의
  • 셸 시안 + IA 트리 + 패널 고정/플로팅/핀 인터랙션
  • 4스쿼드+개발 IA 워크숍(0.5일, 강제 수렴)
⚠️ 의존성 없음 (시작점). S2~S6·DS 트랙의 전제.

SDS 트랙 (토큰·컴포넌트·폴리시) 상시 병행

7월 시드 → 12월 · U1

AC "인라인 hex 금지·CSS 변수, 킷, 콘솔 딥네이비 별도 토큰". 실제 화면에서 쓰이는 것 기준으로 킷이 결정되므로 앞에서 완성 불가 — 시드 후 계속 확장.

  • 7월 시드 — 기존 SDS 토큰 재고, 콘솔 딥네이비 토큰, 베이스 프리미티브(NavItem/Section/Badge/Button/Card)
  • 상시 확장 — S3/S4/S5에서 실제 등장하는 컴포넌트를 킷으로 승격, 폴리시(토스트·확인·스텝퍼·표 정렬·전환) 점진 확정
  • 코어가 단일 소스로 관리 → 스쿼드는 소비·제안, 중복 정의 금지
⚠️ 의존성: S1 셸 토큰 위. 전 구간 S3·S4·S5가 소비하며 동시에 킷을 키움.

S2. 브랜딩·로그인

설계 2주 · 7월 3~4주 · U1

시각적 완결성·첫인상이 필요해 상시 트랙에서 떼어 2주 집중. 로그인은 온보딩(S3) 진입 관문.

기획 3일디자인 1.5주개발 1주
  • 브랜드 톤·로그인 플로우 정의
  • 로고 alph 심볼·favicon·OG, 로그인 화면
⚠️ 의존성: DS 트랙 토큰. S3 온보딩 전 완료 권장.

S3. 멀티프로덕트 온보딩 + 제품별 스텝

설계 5.5주 · 7월3주~8월4주 · U2

CORE-696 최고 성숙(4단계·SolutionResultCard 16케이스). AC "제품 선택→설정 분기, 완료율, 중단 복귀".

분석 1주기획 2주디자인 2.5주개발 5주
  • CORE-696 점검, 통합 계정 진입 갭, 중단 복귀(sub_status)
  • 4단계 플로우, 제품별 스텝 모듈화, 케이스 매트릭스, 화면정책서
  • 전체 화면 세트 + SolutionResultCard 16케이스 + 분기 인터랙션
  • 각 스쿼드 = 자기 제품 온보딩 스텝(위젯·SNS연동) 디테일
⚠️ 의존성: S1 필수, SDS(상시) 컴포넌트 공유. 최대 리스크 → 6주.

S4. 신규 메뉴 (제품 화면군)

설계 5.5주 · 9월1주~10월2주 · U1

통합 콘솔의 본체. AC "제품별 화면군 골격, 공용 컴포넌트 재사용, mock/실데이터 경계". 양 많음 → 스쿼드 병렬.

분석 1주기획 2주디자인 2.5주개발 6주
  • 제품별 화면군 인벤토리(상품·리뷰·메시지·사이트), 공통 패턴 추출
  • 화면군 IA, 제품별 핵심 화면 골격, 공용 컴포넌트 매핑
  • 제품별 화면군 (코어 골격 + 스쿼드 제품 디테일 병렬)
⚠️ 의존성: S1 IA + SDS(상시). 4Q 알파카가 호출하므로 확장 가능 구조로.

S5. 인사이트 표면

설계 6주 · 9월3주~10월4주 · U3

운영자가 통합 콘솔 안에서 보는 데이터 화면. 별도 도구 없이 콘솔에서 통계.

분석 1.5주기획 2주디자인 2.5주개발 5주
  • 외부 분석 임베드 경계(mock/실데이터), 샵 컨텍스트, 행수준 보안, 7개 화면 IA
  • 통계+KPI 드로어+세그먼트(RFM)+코호트+여정+퍼널+인박스
  • 7개 화면 + 차트 컴포넌트(크로스 툴팁) + 드로어 + 인박스 카드
⚠️ 의존성: S1 IA + SDS(상시), 외부 분석 임베드(개발). S4와 겹침 → 골격 8월 말 당겨 분산.

S6. 튜토리얼 + 10분 초기세팅

설계 4주 · 11월 1~4주 · U2

다 만든 뒤 "처음 쓰는 법"을 입힘. AC "인앱 가이드/오버레이, 진행추적" + "10분 내 전제품 기본세팅".

분석 4일기획 1.5주디자인 2주개발 3주
  • 제품별 첫 진입 가이드 시나리오, 10분 세팅 핵심설정 vs 추천기본값
  • 튜토리얼 패턴(코치마크/오버레이/체크리스트), 10분 위저드·타이머 UX
  • 튜토리얼 오버레이 + 제품별 시나리오 + 초기세팅 위저드
